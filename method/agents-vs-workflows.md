# method/agents-vs-workflows — 워크플로 vs 에이전트 · 병렬화 · 검증

> 근거: Anthropic(1차)·MAST(피어리뷰)·Willison(실무) — 4차 리서치 검증.

## 단순함부터
- **가장 단순한 해법**(프롬프트+평가)부터, 명백히 모자랄 때만 다단계.
- **워크플로** = 정해진 코드 경로 / **에이전트** = LLM이 스스로 경로·도구 결정. 자율성↑ = 비용·오류↑ → 가드레일 안에서.

## 멀티에이전트는 언제 (이득/손해 + 비용)
- **이득:** breadth-first·병렬화·단일 컨텍스트 초과·도구 많은 탐색/리서치.
- **손해:** **대부분의 코딩**(병렬화 적고 실시간 조율이 약함).
- **비용:** 단일 에이전트 챗 대비 **~4x 토큰**, 멀티 **~15x**, 토큰량이 성능변동의 80% 설명(BrowseComp). → **고가치 작업만.**
- ⚠️ **더 많은 에이전트 ≠ 더 좋음(MAST):** 이득 종종 미미, 실패 14모드/3범주(명세·설계 / 에이전트 간 정합 / 검증·종료). **기본은 단일 에이전트**, 정말 병렬적일 때만 멀티.

## 오케스트레이터-워커 패턴 (쓸 때)
- 리드가 계획·위임, 서브에이전트는 **병렬 실행이나 동기 대기**.
- 각 서브에이전트엔 **목표 · 출력형식 · 도구/출처 안내 · 명확한 경계** 부여.

## 병렬화 (플레이북 ③기둥, 4시간)
- **2인 분업:** A=빌드 드라이빙 / B=데이터·정확성·발표.
- **서브에이전트:** 조사·정확성 리뷰를 별도 컨텍스트로(메인 오염 없이). 빌드(메인) + 리뷰(서브) 병렬.

## 검증 규율 (바이브 엔지니어링)
- 무책임한 '바이브코딩' vs 책임지는 '엔지니어링'을 가르는 건 **검증·책임**. AI는 기존 역량을 *증폭*.
- **자동 테스트가 단일 최대 enabler**, **test-first/red-green**이 자기수정 루프.
- ⚠️ 주의: AI생성 테스트는 *거짓 자신감*(의도된 동작을 검증해야) · 순진한 TDD 프롬프트는 회귀↑ · **단위테스트만으론 불충분 → 브라우저/결과 수준 검증**(→ `claude-craft/verification-loop`).

# 데이터 어떻게 모으나 (수집 가이드 + AI 인스트럭션)

> 미리 데이터를 쌓아두자는 게 아니다. **필요할 때 빠르게 찾아 '고정 데이터'로 만드는 방법**을 둔다.
> 원칙: 실시간 API 의존 ❌ → 좁게 큐레이션한 **고정 데이터** + **공식 출처 링크** + "지어내지 말 것".

## 1. 어디서 찾나 (공개 출처 우선)
| 분야 | 출처 |
|---|---|
| 통합 공공데이터 | 공공데이터포털 `data.go.kr` · 국가통계 `kosis.kr` |
| 복지·혜택 | 복지로 `bokjiro.go.kr` · 정부24 **보조금24** `gov.kr/portal/rcvfvrSvc`(중앙+지자체 거주지 맞춤) · 한국사회보장정보원 |
| 재난·안전 | 국민재난안전포털 `safekorea.go.kr` |
| 민원·행정 | 국민신문고 `epeople.go.kr` · 정부24 |
| 지역 | 지자체 열린데이터(예: 서울 `data.seoul.go.kr`) · 자치법규/구별 조례 `easylaw.go.kr` |
> 더 찾기: "○○ data.go.kr", "○○ 공식 사이트", "○○ 신청 방법" 검색 → **.go.kr/공식 도메인 우선**.

### 1-b. 빠르게 확보(속도가 경쟁력) — 벌크 먼저, 그다음 좁혀서 채움
**자료를 얼마나 빨리 확보하느냐가 곧 경쟁력.** 순서를 지키면 4시간 안에 충분하다.
1. **벌크 API/파일 먼저**: 공개 OpenAPI·파일데이터로 *목록 전체*를 한 번에 받는다. (예: `data.go.kr` 한국사회보장정보원 **중앙부처복지서비스**·**지자체복지서비스** OpenAPI + 복지서비스정보 파일데이터 — 회원가입·활용신청 보통 자동승인). → 전국·시도 단위를 빠르게 고정 데이터로.
2. **AI 큐레이션으로 좁혀 채움**: 공식 URL을 주고 WebSearch/Fetch로 스키마(§2)를 채운다. *좁게*(1 페르소나·5~30건)부터.
3. **표본 검증**: `source_url`을 실제로 열어 확인(**출처≠검증**). 불확실은 `confidence:"확인 필요"`.
4. **롱테일은 라우팅**: 시군구·동 등 다 못 담는 건 큐레이션 말고 공식 포털(보조금24 등)로 넘긴다(③ 참조).

## 2. AI에게 시키는 법 (복붙 인스트럭션)
```
역할: 공공서비스 데이터 큐레이터.
대상: {분야/시나리오}. 공식 출처(.go.kr 등)만 사용.
작업: 아래 스키마로 5~30건을 JSON 배열로 정리.
규칙: 사실에 없는 건 비우고 confidence="확인 필요". 각 항목 source_url 필수. 지어내기 금지.
규칙(중복 제거용): documents는 배열, 같은 서류는 동일 표기로 통일(자유 서술 금지·부연은 note로). tags는 포함관계를 모두 표기(영아 항목도 ["영아","영유아"]).
스키마: { name, target, benefit, how_to_apply, documents[], tags[], source_url, confidence }
```
- 웹 검색 가능한 도구면 위 인스트럭션 + "공식 출처 우선" → 결과를 `*_sample.json`으로 저장.
- 검증: 표본 몇 건의 `source_url`을 실제로 열어 사실 확인(출처≠검증 — `lessons/ai-chat`).

## 3. 큐레이션 절차 (4시간용)
1. **좁게** — 한 시나리오/1 페르소나로 한정(전부 모으려 하지 말 것).
2. **5~30건**만 — MVP엔 충분. 부족하면 나중에 추가.
3. **스키마 고정** — `welfare_sample.json` 형식 재사용(분야별 파일 추가).
4. **출처·confidence 병기** → 화면에서 링크·신뢰도 표시.

## 4. 형식
- `data-patterns/<domain>_sample.json` (예: `welfare_sample.json`, `disaster_sample.json`)
- 스키마는 `README.md` 참조. 모듈(`patterns/`)이 이 데이터를 코드로 필터 → LLM이 우선순위·설명·결과물.

## 5. 도그푸딩 교훈 (실제로 만들어보고 깨진 것)
> "우리 가족 맞춤 혜택 찾기"(벌크 서류 처리)를 플레이북대로 만들다 데이터 단계에서 드러난 두 함정. **수집할 때 미리 막아라.**

### ① 묶음/중복 제거를 쓸 거면 — **값 표준화(controlled vocabulary)**
- 증상: `"통장 사본"`과 `"아동 또는 부모 명의 통장 사본"`이 **같은 서류인데 문자열이 달라** 중복 제거(`Counter`/union)가 실패. "N종만 준비" 수치가 부풀려짐.
- 규칙: 합치거나 셀 필드(`documents`, `tags`, 카테고리 등)는 **자유 서술 금지 → 사전에서 고른 표준값**만. 부연은 `note`로 분리.
  - ✅ `"통장 사본"` (+ `note: "아동 또는 부모 명의"`)  ❌ `"아동 또는 부모 명의 통장 사본"`
- AI 인스트럭션에 추가: *"`documents`는 아래 표준 서류명 사전에서만 고를 것. 사전에 없으면 추가 후 그 표기를 재사용."* → 파일 상단에 `_vocab` 목록을 두면 AI가 일관되게 채운다.

### ② 자격 힌트 태그는 — **포함관계(계층)를 데이터로**
- 증상: `영아 ⊂ 영유아`인데 단일 평면 태그라, 사용자가 `영유아`만 고르면 영아 전용(부모급여·첫만남)이 **누락**.
- 규칙: 겹치거나 포함되는 분류는 (a) 입력에서 상위 선택 시 하위를 함께 매칭하거나, (b) 데이터에 `tags: ["영아","영유아"]`처럼 **해당되는 상위 카테고리를 모두 적어** 평면 교집합으로도 잡히게 한다.
- 점검 질문(수집 전): *"이 분류들 사이에 '~는 ~의 일종' 관계가 있나? 있으면 한쪽만 골라도 다른 쪽이 매칭되나?"*

### ③ 롱테일·지역 데이터는 — **계층으로 담되, 안정적인 수준까지만 + 나머지는 라우팅**
- 증상: 복지는 **시/도뿐 아니라 시·군·구마다 또 갈림**(서울 25개 구 중 5곳만 출산지원금: 강남 200만 vs 다수 0원 / 시도 위로도 서울 100만·경기 50만·인천 i+1억). 보조금24 **지자체 혜택만 6,774개** — 손으로 다 못 담는다.
- 핵심: **입도(granularity)마다 큐레이션 비용 절벽**이 있다. → **데이터가 안정적인 수준(전국·시도·대표 시군구)까지만 담고, 그 아래(동·미수록 구)는 가짜로 만들지 말고 "거주지 기반 라우팅"**(보조금24·지자체 링크)으로 넘긴다.
- 구현 힌트:
  - 항목 `region`을 **계층 문자열**로(`"서울"` / `"서울 강남구"`, 없으면 전국). 매칭은 **상위 prefix**: 거주지 `"서울 강남구"`는 `전국 + "서울" + "서울 강남구"`를 모두 받고, `"서울 송파구"`는 강남 전용을 안 받음.
  - 입력은 P2 패싯(드롭다운/지도) + 한 단계 드릴다운(시도→시군구). 미수록 지역은 **공식 포털 링크로 정직하게**(가짜 완전성 금지).
- 점검 질문: *"이 도메인은 어느 입도까지 값이 갈리나? 어디까지 담고(전국/시도/대표 시군구), 그 아래는 어떤 공식 포털로 라우팅할까?"*

> 세 교훈의 공통점: **묶기/필터를 코드로 하려면 데이터의 '값'이 먼저 깨끗해야 하고(①②), 다 담을 수 없는 영역은 정직하게 포인터로 넘긴다(③).** 표준화·계층·경계설정은 수집 시점 비용이 가장 싸다.

> ⚠️ 큐레이션 수치·내용은 **사용 전 공식 출처로 재확인**.

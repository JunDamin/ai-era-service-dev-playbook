# InputPatterns — 입력/상호작용 패턴 카탈로그 (맥락별 처방)

> **왜 이 파일:** 입력이 흔들리면 결과가 흔들린다. "한 방에 그럴듯한 디자인"을 뽑지 말고,
> **옵션 유형·맥락을 먼저 분류 → 정해진 패턴을 처방**해 입력을 *안정적·재현가능*하게 만든다.
> `data-input.md`(폼 기본기)·`defaults.md`(기본값/점진적 공개)의 상위 — *무엇을 언제 쓰나*를 고른다.

## 0. 먼저 이 결정표 (맥락 → 패턴)
다섯 가지만 보면 패턴이 거의 정해진다.

| 질문 | 그렇다 → | 패턴 |
|---|---|---|
| 옵션이 **소수(≤7)·서로 독립**인가? | 바로 보여주기 | **P1 평면 컨트롤** |
| **큰 목록**을 독립 속성으로 좁히나? | 속성별 필터 | **P2 패싯 선택** |
| **대부분 기본값으로 충분**, 고급은 가끔? | 숨겼다 펼치기 | **P3 기본값+점진적 공개** |
| 입력이 **길거나 순서가 있고**, 사용자가 답을 더듬나? | 한 질문씩 | **P4 위저드/대화형** |
| 자격/경로가 **복잡·서로 의존**(분기)하나? | 가지 타기 | **P5 결정트리·상황 트리** |
| **롱테일**이라 정형 옵션으로 못 담나? | 자연어 탈출구 | **P6 AI 자유서술→구조화** |

> 대개 **조합**한다. 예) 복지 혜택 = P5(상황 트리)로 자격 가지 + 그 안에 P3(기본값) + P6(직접 쓰기 탈출구).
> 규칙: **선택 > 입력**을 항상 우선(Click > 다중선택 > 짧은 텍스트 > 자연어). 자연어는 보조/탈출구.

## 1. 패턴별 처방

### P1 · 평면 컨트롤 (라디오·체크박스·드롭다운)
- **언제:** 옵션 소수·독립. 라디오=택1, 체크박스=0~다수, 드롭다운/리스트박스=옵션 많아 공간 절약.
- **컨트롤 고르기(NN/g):** 택1·소수 → 라디오 / 다중 → 체크박스 / 옵션 많음·단일 → 드롭다운 / 옵션 많음·다중·공간 여유 → 리스트박스.
- **주의:** 라디오는 항상 1개 선택 강제됨(되돌리기 불가) — "해당 없음"이 필요하면 체크박스. 지표 과다(별표+빨강+인라인) 금지.

### P2 · 패싯 선택 (필터)
- **언제:** 대량 목록을 **독립 속성**(지역·유형·금액 등)으로 좁힐 때.
- **어떻게:** *의미 있는 비율이 실제로 쓰는* 패싯만 둔다 / 자주 쓰는 걸 위로 / **결과 수(카운트) 표시**로 빈 결과 예방 / 맥락 따라 무관 패싯 숨김 / 안 쓰는 패싯은 주기적 제거.
- **주의:** 자격이 **속성 간 의존**이면 패싯이 아니라 P5(트리). 패싯은 "독립 AND"에 강함.

### P3 · 기본값 + 점진적 공개
- **언제:** 대부분 사용자는 표준 경로, 소수만 고급 옵션.
- **어떻게:** 합리적 **기본값**으로 안 건드려도 작동(EAS: 제거→자동화→단순화) / 고급은 접어두고 펼치기 / **공개 단계는 ≤2** (3단계↑면 길을 잃음 → 설계를 단순화하라).
- **주의:** "자주 필요한 건 1차 화면에 다" + "1차에 너무 많이 X" 사이 균형. → `defaults.md`.

### P4 · 위저드 / 대화형 (한 질문씩)
- **언제:** 입력이 **길거나(5+)·순차적**이고, 사용자가 답을 즉시 모를 때. 한 화면 한 과제 → 완료율↑·오류↓.
- **어떻게:** 단계당 한 질문/그룹, 진행·남은 단계 표시, 뒤로/수정 허용. 답은 **버튼/칩**으로(타이핑 최소).
- **주의(중요):** *입력값이 정해져 있고 구조가 중요하면 정형 폼이 채팅보다 빠르고 명확*하다. 대화형은 **탐색·의사결정이 어려울 때** 빛난다. 순수 챗봇은 매 턴 지연·비용·헛질문 위험 → 기본값 아님.

### P5 · 결정트리 / 상황 트리 (분기·의존)
- **언제:** 자격/경로가 **복잡하고 서로 의존**할 때(복지 자격, 진단, 견적). "조건을 넣으면 해당 항목이 정해진다."
- **어떻게:** 해당되는 **가지를 켜고**(다중) → *자격을 실제로 가르는 한 단계만* 드릴다운 → 가지 아래 항목을 **전부 수집**. 무관한 질문은 안 보이게(skip/jump). 분기 조건은 **제한적인 것 먼저** 평가.
- **워크드 예시(도그푸딩):** 복지 = [영유아·아동][임신·출산][어르신][한부모][맞벌이]… 가지 → 영유아 안에서 *보육형태(가정양육↔보육료 택1)*만 더 물음 → 켜진 잎을 union. 가지↑ → 항목↑ → 공통 서류↑(벌크 가치↑). **검증됨:** 한부모+육아를 같이 켜면 사회보장급여·소득재산·통장·신분증이 **가지를 넘어 공유**돼 통합 준비물이 더 강해짐.
- **주의:** 트리는 강력하지만 과설계 위험. **가르는 노드만** 두고, 데이터에 분류의 **포함관계(계층)**를 반영해야 평면 매칭도 샌다(→ `../data-patterns/how-to-collect.md` §5).

### P6 · AI 자유서술 → 구조화 (탈출구)
- **언제:** 정형 옵션으로 못 담는 **롱테일**, 또는 "그냥 한 줄로 쓰고 싶다"는 사용자.
- **어떻게:** 자연어 입력 → LLM이 **구조화 필드로 파싱** → 결과를 **수정 가능한 칩/체크로 되돌려 사람이 확인·수정 후 확정**(검토 없는 자동 채움 금지). 파싱 실패·모호 → 정형 입력으로 폴백.
- **주의:** 절대 *유일 입력 경로로 두지 말 것*(선택>입력 위반). 실시간 외부 API 자동조회는 데모/MVP에서 비목표.
- **실전 검증(도그푸딩):** 단순/키워드 파서는 **부정·모호**를 놓친다 — "어린이집 *안* 보내요"→이용으로 오인, "7살"→연령 오매핑. 교훈: **P6의 안전은 파싱 정확도가 아니라 "수정 가능한 컨트롤로 환원 → 사람이 확정"에 있다.** LLM을 쓰더라도 마지막 확인·되돌리기는 사람 몫. (그래서 P6는 항상 P1~P5 위에 얹는 *탈출구*로 설계)

## 2. 횡단 규칙 (어떤 패턴이든)
- **영구 레이블**(placeholder를 레이블로 X) · **선택>입력** · **검증은 on-blur**(타이핑 중 조기검증=적대적) · 제약 조건은 미리 고지.
- **빈 결과 예방:** 카운트/미리보기, empty state에 예시·다음 행동.
- **표준값(controlled vocabulary):** 묶음/중복 제거를 코드로 하려면 입력·데이터 값이 표준화돼야 함(→ data-patterns §5).
- **AI 출력엔 confidence + 출처**, 사람이 마지막에 확인.

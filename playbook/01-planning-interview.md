# 01 · 기획 인터뷰 (함께 기획 · 상호작용)

> **목적:** 바로 코딩하지 말고, AI가 사용자와 **대화로 기획을 좁힌다.** 결과는 `02-project-template.md`를 채운 기획서 초안.

## AI 진행 규칙 (Claude가 따른다)
- **질문은 하나씩**, 답을 기다린 뒤 다음으로. (한 번에 쏟지 말 것)
- 각 질문에 **추천답(1안)을 먼저 제시** → 사용자는 고르거나 고친다. (빈 화면 금지 = 인터뷰에도 적용)
- 답이 모이면 즉시 `02-project-template`의 해당 칸을 채워 보여준다.
- 모호하면 **구체 시나리오/예시로 되묻는다.** 범위가 커지면 "그건 non-goals(v2)로 둘까요?"라고 막는다.

## 질문 순서 (각 답 → 템플릿 칸)
1. **누구의 문제인가?** 구체적 1인(이름·상황) → §2 대상
2. **무엇이 문제인가?** 한 문장 → §1 문제
3. **지금은 어떻게 하나?** 현재 흐름·시간 → §3 현재 흐름
4. **어디서 막히나?** 가장 아픈 1개 → §4 Pain
5. **가장 작은 가치 단위는?** "[대상]이 [과업]을 [결과물]로 끝낸다" → §5 핵심 가치
6. **★ UX 철학 — 사용자는 어떤 상태로 도착하나?** (감정·상황·디지털 친숙도: 불안한 임차인? 잠 못 잔 보호자?) 그리고 **첫 화면 10초에 '무엇 하나'를 이해시켜야 하나?** 그 단순함을 위해 **무엇을 숨기거나 대신해 줄 것인가?** → §5 서비스 철학
   - *왜 묻나:* 성공한 서비스는 기능이 아니라 철학으로 갈린다(토스 Simplicity — "1초 만에 이해, 배울 것 없음"). 이 답이 입력 패턴·라이팅 톤·화면 구성을 전부 결정한다.
7. **절대 만들지 않을 것은?** (로그인·권한·다국어·결제 기본 제외) → §6 MVP/non-goals
8. **입력·출력은?** 선택>입력 / 결과물 형태 → §7 입출력
9. **틀리면·비면·느리면?** 실패 시나리오 → §9 실패
10. **성공 기준(끝)은?** pass/fail 1개 → §10 검증

## 마무리 (인터뷰 종료 시)
- 채워진 §1~§10을 **기획서 초안**으로 요약 제시 → 사용자 승인.
- ★ **해피패스 유저저니·스토리보드 5칸**을 정리한다(텍스트/ASCII로 충분): 도착 상태 → 단계별 '무엇이 뜨나·사용자가 무엇을 하나' → **완주**. 이 하나가 *사람↔AI 공용 기준*이자 네 역할을 겸한다 — **정렬**(빌드 브리프) · **검증**(완주=§10 pass/fail) · **발표**(해결 후 모습) · 데모 대본. → §5b
- 승인되면 **§11 AI Coding Instructions**(개발 지시서)를 생성하고 빌드로 넘어간다.
- 빌드 전 체크: `lessons/README.md` 항상-적용 체크리스트.

> **와이어프레임·정보구조도(IA)·플로우차트는 따로 그리지 않는다** — 패턴(A/B/C/D)+템플릿+`tokens.css`가 그 기본값이고, AI가 회색 박스보다 빠르게 진짜 화면을 뽑는다(AI가 산출물 사다리를 무너뜨림). *문제 고유*한 건 해피패스 스토리보드 하나뿐 — 그것만 만든다.

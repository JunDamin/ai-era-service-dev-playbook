# 의사결정 (Decision, Type B)

> **한 줄:** 상황을 받아 위험도·우선순위를 판단하고 **행동 계획**으로 연결한다.

## ★ B형 고유 원칙 (전세 위험 점검 드라이런으로 검증)
> 추천형(A)을 그대로 베끼면 B는 망가진다. B에서만 통하는 4가지를 먼저 박아라.

### ⓐ '모름'의 비대칭 — 미입력=위험 가산 (B의 심장)
- **추천형(A)은 미입력=필터에서 제외**(모르면 후보를 줄임). **위험형(B)은 정반대 — 미입력=위험으로 가산**한다. *"확인 안 한 것 자체가 리스크"* 이기 때문.
- 답을 `예/아니오/**모름**` **트라이스테이트**로 받고, `모름`에 **0이 아닌 위험점수**(보통 적신호의 60~70%)를 준다.
- 효과(실측): 전세 점검에서 **'전부 모름' = 11점**(위험 높음)이 **알려진 깡통전세 10점보다 높게** 나옴 → "아무것도 확인 안 한 상태가 가장 위험"이 점수로 증명됨. 사용자를 *확인 행동*으로 민다.
- ❌ 안티패턴: 모름을 0점·중립으로 처리 → 게으른 사용자가 "안전"으로 오인(거짓 안심). B에서 가장 위험한 버그.

### ⓑ 판정 회피의 UX — 못 판정해도 '행동'은 준다
- B는 법·의료·자격을 **단정하면 안 된다**(`confidence='확인 필요'`). 그런데 Rule④는 결과물을 요구한다 → 긴장.
- 해소: **확인처로 라우팅**한다. 신호별로 *모름→확인 행동*(어디서 확인할지 링크), *적신호→대응 행동*(구체적 다음 수)으로 변환해 우선순위 카드로.

### ⓒ 신호 가중합의 투명성 — 점수 말고 '기여'를 보여라
- 종합 점수만 주면 불신·블랙박스. **신호별 +기여를 펼쳐** "왜 이 색인지" 보이게(→ `lessons/dashboard.md` §6).

### ⓓ 데이터 사각지대 = 안전 메시지로 전환
- 자동조회가 안 되는 영역(예: 빌라 시세 불투명)은 한계가 아니라 **메시지**다 → "그래서 직접 확인해야 한다"(ⓐ와 맞물림). 수집/라우팅은 → `data-patterns/how-to-collect.md` §5-④.

## 언제 쓰나
- 재난 대응, 위험도 분석, 전세·계약 위험 점검, "지금 어떻게 해야 하나"류

## 입력 계약
- 선택형 우선: 지역 · 상황 유형 · 조건(예: 가족 구성). 받은 값은 `session_state` 저장.
- **위험 신호는 `예/아니오/모름` 트라이스테이트**(ⓐ). 가능하면 식별자(주소·유형)로 **고정 데이터에서 자동채움** → 없으면 모름 + 확인처(ⓓ).

## 처리 (파이프라인 내 역할)
- Context Extraction: 입력 → 상황 프로필
- 위험 평가: 고정 규칙/데이터 기반 위험도 산정(코드) + LLM 설명. **신호별 가중치 합산**, `모름`도 가산(ⓐ).
- 우선순위 + 행동 계획 도출(모름→확인행동, 적신호→대응행동; ⓑ)

## 산출물 (Artifact)
- **위험도(시그널 색)** · **신호별 기여**(왜 이 색인지, ⓒ) · **행동 계획 카드**(지금 할 일 순서, 점수순)
- 조합: **+ Type C**(체크리스트) → 준비 항목까지 · **+ Type D**(리포트 다운로드)

## Claude 프롬프트 (복붙)
```
<instructions>
상황으로 위험도(High/Med/Low)와 우선순위, 즉시 행동을 제시.
- 단정적 '판정'(의료·법률) 금지. 불확실하면 confidence='확인 필요'.
- '모름/미확인' 신호는 안전이 아니라 위험으로 다뤄라(확인 전까지 위험). 미확인 항목은 '확인 행동'으로 변환.
- data 밖은 지어내지 말 것.
</instructions>
```

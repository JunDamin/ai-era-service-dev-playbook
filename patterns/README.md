# patterns — 조합 가능한 기능 모듈

> **서비스 유형 = 모듈.** 문제 = 척추(Input → Context → Discovery → Recommendation → Explanation → Action → Artifact)에 모듈 **1~N개를 조합**. 대부분 2개 조합.
> A·B·C·D는 **도메인 무관 파이프라인 모듈**이다(복지 전용 아님). 아래 예시가 복지·전세·출산에 쏠린 건 *실제로 그 분야로만 드라이런했기 때문* — 안 해본 분야 예시는 지어내지 않는다(검증된 것만).

| 유형 | 정의 문서 | 보일러플레이트 |
|---|---|---|
| **A 추천/매칭** | `recommendation.md` | `../stack/templates/recommend_app.py` |
| **B 의사결정/위험** | `decision.md` | `../stack/templates/decision_app.py` |
| **C 체크리스트/준비** | `checklist.md` | `../stack/templates/checklist_app.py` |
| **D 문서 생성** | `document.md` | `../stack/templates/document_app.py` |

- 조합 예: 혜택 찾기 = **A + D** · 임산부 원스톱 = **C + A** · 재난 가이드 = **B + C** · 생활 불편 = **A + D**
- 새 모듈은 `_TEMPLATE.md` 복제(입력계약→처리→산출물→조합→프롬프트→데이터→근거).

## ★ 빈칸 비대칭 — 같은 '미입력'인데 유형마다 처리가 정반대다
> 입력을 설계하기 전에 반드시 정하라: **사용자가 답을 안 했을 때(모름·미해당·빈칸) 무슨 일이 일어나는가?** 드라이런 3종(복지 A · 전세 B · 출산 C)에서 모두 검증.

| 유형 | 미입력의 의미 | 처리 | 틀리면 생기는 일 |
|---|---|---|---|
| **A 추천** | 조건 불명 | **후보 제외**(보수적 축소) 또는 전체 유지+확인 라벨 | 과잉 추천(자격 없는 것 노출) |
| **B 위험** | **확인 안 함 = 그 자체가 리스크** | **위험 가산**(모름≠0점, 트라이스테이트) | `모름=0` → 거짓 안심(가장 위험한 버그) |
| **C 체크리스트** | 해당 없음 | **항목 숨김**(분모도 함께 축소) | 무관 항목 노출 → 리스트 불신·이탈 |
| **D 생성** | 필수 사실 누락 | **갭점검 경고**(생성은 하되 빈칸 `[__]` 표시) | 지어내기(AI 슬롭)·반려 |

→ 상세: `decision.md` ⓐ(모름=가산) · `checklist.md` ⓒ(조건부 숨김) · `document.md` ⓑ(갭점검)

## 환류 관례
- 드라이런에서 나온 교훈은 **"조건 → 처방"** 형식으로 해당 모듈 문서에 + **도메인을 지워도 남는 메커니즘이면 해당 템플릿(.py) 기본값에도** 함께 반영한다. (드라이런 코드 이식 ❌ — 교훈만)

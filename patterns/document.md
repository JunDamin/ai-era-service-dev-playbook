# 문서 생성 (Document, Type D)

> **한 줄:** 입력을 받아 **민원·신청서·보고서 초안**을 생성한다.

## ★ D형 고유 원칙 (생활 민원 라우터 드라이런으로 검증)
> **생성형의 해자는 '글'이 아니라 그 주변이다.** 텍스트 생성만 하면 "그냥 입력 폼"과 다를 게 없다(실제 첫 버전이 그랬다). 가치는 생성 전후 단계에서 나온다.
### ⓐ 라우팅 — 사용자가 '유형'을 몰라도 되게
- 문서 유형/접수처가 여럿이면: 유형을 고르게 하지 말고 **상황 서술로 자동 분류**(LLM+키워드 폴백)하고, 결과를 **수정 가능한 선택**으로 보여줘라. "어디에 넣을지 모름"이 보통 진짜 문제다.
### ⓑ 갭점검 — 생성 전에 빠진 필수 정보를 잡아라
- 유형마다 **필수 정보 목록(`needed[]`)** 을 데이터로 두고, 생성 전 입력과 대조해 **빠진 항목을 경고**(반려·재작성 예방). 점검은 코드로 — 전용 입력칸(위치·일시)이 있으면 그 칸 기준으로. (미입력 처리는 유형마다 다름 → `README.md` 빈칸 비대칭 표)
### ⓒ 효과화 — 템플릿 빈칸 채우기는 AI 슬롭
- 초안은 **사용자가 준 사실만으로** 효과적 프레이밍(공익 영향·반복성·구체적 요구사항)을 입혀라. 없는 사실·법조항·수치 지어내기 금지, 산출물엔 "보조 도구 생성물, 제출 전 확인" 고지.
### ※ 입력 예외 — D형은 **자유 서술이 주입력**
- 추천형의 "선택>입력"과 달리, 생성형은 상황 서술 자체가 원료라 **자연어가 1차 입력**이다(→ `lessons/input-patterns.md` P6). 구조 정보(위치·일시)만 전용 칸으로 분리.

## 언제 쓰나
- 민원 초안, 신청서 작성, 보고서 초안류

## 입력 계약
- **자유 서술(주입력)** + 구조 칸(위치·일시 등) + 유형은 자동분류 후 수정 가능(ⓐ). `session_state` 저장.

## 처리 (파이프라인 내 역할)
- Context → **유형 분류(ⓐ)** → **갭점검(ⓑ)** → 사실 기반 초안 생성(ⓒ, 편집 가능)

## 산출물 (Artifact)
- **문서 초안(편집 가능 텍스트)** + 복사 + 다운로드(.md)
- 조합: **A/B/C의 결과를 문서로** 마무리(예: 추천→신청서)
